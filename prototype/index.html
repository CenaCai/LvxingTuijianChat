1. 产品定位

「智行 TravelMind」是基于 LLM 的全流程 AI 差旅助手。CUI 对话 + GUI 卡片混合交互，覆盖"灵感激发 → 规划决策 → 行程守护"完整链路。

市面上现有平台多为"单点能力"：OTA AI 负责问答推荐、企业差旅负责预订报销、TripIt/Wanderlog 负责行程管理、航司系统负责异常恢复。我们的差异化是把这些能力整合成一个可解释、可确认、可执行的 Agent 工作台。

3. 核心差异化

  • 全链路整合 — 灵感（类似 GuideGeek）→ 决策（类似 Google Travel Canvas）→ 记忆（向量数据库+治理）→ 执行（类似 Navan MCP 编排）→ 异常恢复（类似 AIRS + LLM）
  • 可解释 Agent — 每次推荐标注数据来源(souce)+查询时间(fetched_at)+置信度+不确定性声明，类似 GuideGeek 的人工审核思路但用自动化实现
  • 长记忆系统 — 独立于 LLM 上下文窗口的向量记忆，跨行程复用，第45天记得第1天的偏好
  • MCP 工具编排 — 通过 MCP 协议标准化接入携程/飞猪/天气/航班/知识库等外部服务，Agent 按依赖关系自动编排调用链
  • 双通道风险感知 — 系统主动监控（类似 FlightSense 预测）+ 用户手动上报，确保异常不遗漏

4. 三大用户场景

A · 长线游学知识库

用户正规划为期2个月的欧洲游学，前期收集了大量小红书攻略、签证政策、个人学习计划。痛点：碎片信息难以结构化；第45天时无法回忆起第1天设定的饮食偏好或预算限制。AI方案：用户粘贴碎片→自动提取实体+约束→存入向量数据库→第45天自动激活相关记忆→主动提醒"根据Day1设定，你偏好清淡饮食"。

B · 复杂行程对比决策

用户在"新疆15天自驾"和"云南15天游学"间纠结。痛点：信息分散（天气、机票、住宿、课程评价），多维度难以量化比较。AI方案：并行调用4个MCP Server→6维评分矩阵→用户调节权重实时重算→展示每项评分的数据来源和不确定性。

C · 行程突发与动态调整

旅行途中突发状况（景点关门、极端天气航班取消）。痛点：手动查询备选方案耗时；调整涉及多个环节（课程/交通/住宿）。AI方案：天气MCP 15分钟轮询→自动检测异常→Skill: generate_plan_b 生成2-3个替代方案→用户确认→Skill: execute_rebook 编排MCP链一键执行。

7. 防幻觉机制（5道防线）

  • ① 强制溯源 — 所有数据类声明（价格、天气、时间）必须携带 source 和 fetched_at 字段，无来源=不可信
  • ② 置信度门槛 — 意图识别 confidence <0.7 时强制追问用户；工具调用结果标记置信度
  • ③ 数值区间保护 — 工具返回的价格区间 [¥1600, ¥4200] 不可被模型缩小为"大约¥3000"
  • ④ 未知即声明 — 无法获取的数据标注为"暂未获取"而非编造默认值，类似 GuideGeek 人工审核思路
  • ⑤ 人工确认卡点 — 涉及支付/退款/隐私的操作，Agent 不可自动执行，必须用户逐项勾选确认

9. 设计阐述

为什么 CUI + GUI？对话（CUI）降低表达门槛——用户不需要填表单，像跟朋友聊天一样描述想法；结构化卡片（GUI——对比矩阵、时间线、编排链）补充纯文本的不足。参考 GuideGeek 的 WhatsApp 轻入口 + Google Travel Canvas 的可视化画布。

LLM 能力边界（做不到什么）

  • 做不到精准数值预测——机票价格/天气必须依赖外部 API，LLM 只负责推理和推荐
  • 长对话记忆衰减——超过 100k token 后对早期信息召回准确率下降，因此独立记忆系统比依赖上下文窗口更可靠
  • 幻觉不可避免但可治理——通过 5 道防线（溯源+置信度+区间保护+未知声明+确认卡点）将幻觉率控制在可接受范围
  • Plan B 不能只靠 LLM 生成文案——背后需要规则引擎+库存/订单工具+人工确认，参考 AIRS disruption recovery

未来演进

  • Agent 自主执行：从"建议"到"执行"，用户授权后 Agent 可自主完成机票/酒店/课程预订
  • 多模态输入：拍照提取签证文件/手写笔记/机票截图→结构化信息
  • 社交协同：多人行程协同规划、费用分摊、偏好融合（参考 Wanderlog）
  • 端侧部署：核心记忆和偏好本地存储，保护隐私+降低延迟
  • 企业版：接入企业差旅政策（参考 Navan/SAP Concur），自动合规校验
